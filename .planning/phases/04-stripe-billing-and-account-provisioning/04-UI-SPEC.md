## Spacing Scale

Mantine uses its own spacing scale internally. All layout decisions in Phase 4 follow the existing Mantine prop conventions used in `content-security.tsx` and `manage-billing.tsx`. Declared values below map Mantine prop names to px equivalents for executor reference.

| Token | Mantine Prop | Value | Usage |
|-------|-------------|-------|-------|
| xs | `xs` | 4px | Icon gaps, inline padding |
| sm | `sm` | 8px | Compact element spacing, Button `size="xs"` padding |
| md | `md` | 16px | Default element spacing (`mb="md"`) |
| lg | `lg` | 24px | Section padding (`mb="lg"`, `Divider my="lg"`) |
| xl | `xl` | 32px | Layout gaps (`gap="xl"`) |
| 2xl | — | 48px | Not used in Phase 4 UI scope |
| 3xl | — | 64px | Not used in Phase 4 UI scope |

Exceptions:
- Auth container top margin: inherited verbatim from `auth.module.css` — not declared or modified by Phase 4; executor must not change these values
- Button action size in admin table rows: `size="xs"` (matches Phase 3 `content-security.tsx` pattern)

## Typography

All type is rendered by Mantine Text, Title, and Button components. Sizes use Mantine's semantic size tokens, which map to the values below (Mantine defaults, unoverridden in `theme.ts`).

| Role | Mantine Prop | Approx Size | Weight | Line Height |
|------|-------------|------------|--------|-------------|
| Body (table cell, description) | `size="sm"` | 14px | 400 | 1.5 |
| Body standard | `size="md"` | 16px | 400 | 1.5 |
| Label / column header | `Table.Th` with `fw={500}` | 14px | 500 | 1.4 |
| Heading / section title | `Title order={2} fw={500}` | 24px | 500 | 1.2 |

Two weights only: **400** (body) and **500** (headings and column headers). `Table.Th` default weight must be overridden with `fw={500}` to eliminate the browser default bold.

Conventions confirmed from `content-security.tsx` and `forgot-password-form.tsx`:
- `Text c="dimmed"` for secondary/help text
- `fw={500}` for heading weight (not 700)
- `Text size="sm" c="dimmed"` for subscription descriptions in ManageBilling pattern

## Color

Mantine handles color via CSS variables with automatic light/dark theming. Phase 4 uses the same semantic color tokens as all existing settings pages.

| Role | Mantine Token | Usage |
|------|--------------|-------|
| Dominant (60%) | `var(--mantine-color-body)` | Page background, container background |
| Secondary (30%) | `var(--mantine-color-default-border)` + card surfaces | Table rows, settings panel background |
| Accent (10%) | `blue[6]` = `#0b60d8` (from `theme.ts`) | Primary Button fills, links |
| Destructive | `red[6]` = `#d43535` (from `theme.ts`) | "Revoke Access" button variant, Badge `color="red"` |
| Status positive | Mantine `green` default | Badge `color="green"` — active status |
| Status warning | Mantine `orange` default | Badge `color="orange"` — locked/payment-failed status |

Accent reserved for:
- Primary action buttons ("Manage Subscription", "Restore Access")
- External link anchor color (Stripe Dashboard link)

Destructive reserved for:
- "Revoke Access" button (admin table)
- Badge indicating locked/cancelled subscriber status

Badge color mapping (source: Phase 3 pattern from `content-security.tsx`):
- Active → `Badge color="green"`
- Locked (billingLockedAt set) → `Badge color="orange"`
- Cancelled → `Badge color="red"`

## Interaction Contracts

### Admin Subscribers Tab

1. Page load: `if (!isAdmin) return null` — non-admin users see nothing (no redirect, no error).
2. Data fetch: `useEffect` calls `GET /api/billing/admin/subscribers` on mount when `isAdmin === true`.
3. Loading state: render `<Text c="dimmed">Loading...</Text>` while fetch in progress.
4. Empty state: render single `<Table.Tr>` with `<Table.Td colSpan={6}>` containing `<Text c="dimmed">No subscribers found.</Text>` — table headers always visible.
5. Revoke Access click: open `modals.openConfirmModal()`. On confirm: `POST /api/billing/admin/revoke` with `{ userId }`. Re-fetch subscriber list on success.
6. Restore Access click: `POST /api/billing/admin/restore` with `{ userId }`. Re-fetch subscriber list on success. No confirmation modal needed (non-destructive).
7. Stripe Customer link: `<Anchor href="https://dashboard.stripe.com/customers/{stripeCustomerId}" target="_blank" rel="noopener noreferrer">View in Stripe</Anchor>`. Kiwify rows: render `—` plain text.

### `/welcome` Page

1. No auth check — public route.
2. Static render only — no API calls, no interactive elements.
3. No loading state needed.

### Customer Portal Button

1. Visible only to users with `role === 'reader'` AND `gateway === 'stripe'` (Kiwify users have no portal).
2. Click: call `GET /api/billing/portal` → on success, `window.location.href = response.url` (redirect to Stripe portal).
3. Error: show Mantine notification — `notifications.show({ color: 'red', message: 'Failed to open billing portal. Please try again.' })`.

### Billing-Locked Error State

1. Triggered when API returns 403 with `error: 'BillingLocked'` in response body.
2. Display varies by gateway detected from user context (stripe vs kiwify).
3. Stripe users: show "Manage Subscription" button calling `/api/billing/portal`.
4. Kiwify users: show static "contact support" text — no button.
